1. Introduction & Objectives

The MediCare Hospital Management System (HMS) is designed to provide efficient data organization, retrieval, and management for a real-world healthcare environment. The integrated database manages core operational functionalities including patient records, doctor scheduling, appointment management, ward/bed allocations, and billing systems.

Key Project Objectives:

  • Streamline the management of hospital resources, including doctors, patients, wards, and beds.
  • Implement a robust normalized database to eliminate data anomalies and inconsistencies.
  • Provide seamless handling of day-to-day operations such as appointment booking, monitoring bed availability, and generating billing.
  • Enforce strict data integrity using primary/foreign keys, domain constraints, and transaction management.
  • Guarantee accurate data retrieval using optimized multi-table JOINs and analytical queries.

2. Database Schema Design

The database schema is defined in MySQL using the InnoDB storage engine to support transaction properties. The primary entities are structured as follows:

  • patients: Demographics and unique token_number identifiers.
  • doctors: Specializations and experience metrics.
  • appointments: Bridge table connecting instances of doctor-patient engagements.
  • wards & beds: Relational tree segmenting infrastructure types with foreign keys dynamically assigning beds to admitted patients.
  • billing & billing_items: Financial hierarchy isolating overall invoices from their detailed line items.
  • users: RBAC authorization table tied to staff identifiers.

Primary Entity Relationships:

  • 1-to-N: A patient can have multiple appointments and billing invoices.
  • 1-to-N: A ward contains multiple beds.
  • 1-to-N: A billing instance acts as a parent to multiple billing_items.
  • 1-to-1: A user credential record maps to a specific doctor identifier.

3. Database Normalization (Up to 3NF)

To avoid insertion, update, and deletion anomalies, the tables were normalized up to the Third Normal Form (3NF):

  • First Normal Form (1NF): All tables hold a defined primary key (id AUTO_INCREMENT). Attributes contain atomic values only. For example, the patients table does not store multiple phone numbers in a single column; it relies on atomic storage variables.
  • Second Normal Form (2NF): Every non-prime attribute is fully functionally dependent on the entire primary key. In the billing_items table, description, quantity, and unit_price depend specifically on the line item id. They do not depend partially on any other composite key.
  • Third Normal Form (3NF): The schema complies with 2NF without any transitive dependencies. Attributes like specialization or experience reside purely in the doctors table rather than repeating within users. A user's access role does not determine medical expertise. Dynamic states such as Confirmed and Pending are strictly typed and don't duplicate relational string values across tables.

4. Integrity Constraints

The following integrity constraints are embedded strictly within the MySQL DDL script to guarantee data validity:

  • Entity Integrity (Primary Keys): id INT AUTO_INCREMENT PRIMARY KEY in all tables guarantees mathematical uniqueness for tuple selection.
  • Referential Integrity (Foreign Keys): Deleting a patient executes an ON DELETE CASCADE which drops associative appointments and billing records. Deleting an admitted patient triggers an ON DELETE SET NULL on their bed array, safely freeing the resource.
  • Domain Constraints (ENUMs): ward_type is restricted to ('General', 'ICU', 'Emergency', etc.), acting as domain constraints.
  • Unique Constraints: Fields like token_number and invoice_number enforce UNIQUE constraints to prevent duplicate external logic.
schema_snippet.sql SQL (DDL) 
CREATE TABLE IF NOT EXISTS appointments (
    id INT AUTO_INCREMENT PRIMARY KEY,
    patient_id INT,
    doctor_id INT,
    status ENUM('Pending','Confirmed','Cancelled','Completed') DEFAULT 'Pending',
    FOREIGN KEY (patient_id) REFERENCES patients(id) ON DELETE CASCADE
);

5. Query Processing & Analytics

The application executes complex SQL queries dynamically to generate dashboards and report analytics:

view_appointments.php - JOIN Query SQL 
SELECT a.*, p.name AS pname, d.name AS dname, d.specialization 
FROM appointments a 
JOIN patients p ON a.patient_id = p.id 
JOIN doctors d ON a.doctor_id = d.id 
WHERE a.status = 'Confirmed' 
ORDER BY a.appointment_date DESC;
reports.php - Analytical GROUP BY Query SQL 
SELECT DATE_FORMAT(created_at, '%b %Y') AS label, COUNT(*) AS cnt 
FROM patients 
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 12 MONTH) 
GROUP BY DATE_FORMAT(created_at, '%Y-%m') 
ORDER BY DATE_FORMAT(created_at, '%Y-%m');

6. Transaction Management

The system utilizes database-level transactions in the InnoDB engine to ensure ACID properties during bulk entries, such as generating an invoice and writing its separate line items:

billing.php - Transaction Logic PHP (PDO) 
$pdo->beginTransaction();
try {
    // 1. Log overall invoice state
    $stmt = $pdo->prepare("INSERT INTO billing (...) VALUES (...)");
    $stmt->execute([...]);
    $billingId = $pdo->lastInsertId();
    
    // 2. Iterate and append segmented invoice data seamlessly
    $iStmt = $pdo->prepare("INSERT INTO billing_items (billing_id, ...) VALUES (?, ...)");
    foreach ($items as $item) {
        $iStmt->execute([$billingId, ...]);
    }
    // Complete the synchronized commit 
    $pdo->commit();
} catch (Exception $e) {
    // Discard any partial inputs upon hardware or logic failure
    $pdo->rollBack();
}
